# MNTS 从项目恢复到纸上观察阶段完整开发总结

## 1. 文档目标

这份文档用于把本轮开发全过程完整沉淀下来，覆盖以下内容：

1. 项目恢复后到底做了哪些关键工作。
2. 系统主线是如何从研究阶段一步步收敛到观察阶段的。
3. 中途遇到了哪些问题，以及分别如何解决。
4. 做过哪些验证、为什么要做、结论是什么。
5. 当前系统已经走到哪一步，还差什么，下一阶段应该怎么看。

这不是一份单纯的“结果报告”，而是一份完整的开发与决策复盘文档。

---

## 2. 当前总状态

截至本文档生成时，MNTS 已经从“项目恢复后的研究与验证阶段”推进到：

- `BTC + ETH`
- `15m`
- `V2 状态层`
- `breakout + V2`
- 固定 anchor 参数
- `180天训练 -> 45天冻结运行 -> 到期再滚动重训`
- `close 发信号 / next open 成交`
- `fee4 + slippage1`
- 本地连续 `paper ledger`
- 动态中文 Web 看板

也就是说，当前已经进入：

- `第一版严格口径纸上观察期`

这不是历史回放展示页，也不是早期轻量实时重训实验版，而是一个尽量对齐历史 walk-forward 回测口径的未来实时纸面观察系统。

## 3. 整体开发路线图

本轮工作的主线可以分成九个阶段。

### 3.1 阶段 A：项目恢复与主线重新确认

这个阶段的目标不是继续盲目扩展，而是先确认当前项目恢复后的“最强主线”是什么。

围绕这个目标，逐步形成了几个关键共识：

- 当前最有价值的主线不是继续发散更多策略，而是把已有优势主线收敛。
- `15m + V2 + breakout + V2` 是最值得工程化推进的方向。
- 参数不应该选历史单点最优，而应选稳定区中心，也就是后面反复提到的 `anchor 参数`。
- `fee4` 代表 `4 bps = 0.04%` 单边手续费，round-trip 约 `0.08%`。

这一阶段的意义在于：

- 把整个项目从“研究探索模式”切换到“实盘前工程收敛模式”。

### 3.2 阶段 B：跨品种泛化验证

确认主线后，并没有马上进入观察，而是先继续验证“这条主线是否只是 BTC 特例”。

当时讨论过两个方向：

1. 验证黄金或黄金风格资产。
2. 验证流动性更低、质量更差的币种。

围绕黄金方向，当时确认了：

- `PAXGUSDT` 是 tokenized gold，不是传统黄金期货本体。
- 它能作为黄金风格资产的近似验证对象，但不等于真正的 COMEX 黄金连续合约。
- 真要做黄金期货，需要额外处理换月、连续合约、数据质量问题，工程复杂度明显更高。

因此后续转向更务实的方案：

- 在币安找时间足够长、成交量中下等的老山寨做验证。

最终完成了：

- `DASHUSDT` 的完整验证。

结论是：

- `breakout + V2` 不只对高流动性主流币有效。
- 在中下流动性老山寨上也仍然有效。
- 但回撤和行为更激进，因此不适合作为第一批收敛对象。

### 3.3 阶段 C：五币种统一统计与比较

在多品种验证后，为了避免“结论散落在多个脚本和输出目录里”，生成了统一的五币种统计表。

覆盖对象：

- `BTCUSDT`
- `ETHUSDT`
- `SOLUSDT`
- `XRPUSDT`
- `DASHUSDT`

输出文件：

- `five_symbol_comparison_4y.md`
- `five_symbol_comparison_4y.csv`

统一口径：

- `4y 15m`
- `V2`
- `breakout + V2 walk-forward`
- `parameter stability`

这个阶段的意义在于：

- 把“反复讨论过很多次的结果”收拢成可对比、可复查、可沉淀的统一基线。

### 3.4 阶段 D：明确系统收敛方案

当时正式写出了收敛方向文档：

- `MNTS_系统收敛方案_实盘前版本.md`

核心结论包括：

- 主线冻结为 `15m + V2 + breakout + V2`
- 第一轮收敛样本优先用 `BTC + ETH`
- 参数使用稳定区 anchor，不追求历史峰值点
- 不再继续大规模扩品种和扩策略家族

这一阶段实际上是项目的分水岭：

- 在此之前，项目主要是“研究证明有没有用”
- 在此之后，项目主要是“把能用的东西压成一套系统”

### 3.5 阶段 E：执行口径从研究模式收紧到更接近实盘

主线收敛后，下一步不是直接上看板，而是先收紧回测执行口径。

原本更偏研究口径的方式大致是：

- 当前 bar 收盘生成信号
- 按 close 附近成交

后续收紧为：

- `close 发信号`
- `next open 成交`
- `fee4 + slippage1`

并重新补跑了 `BTC / ETH` 的主线和参数稳定性。

结论是：

- 指标强度有下降，这是正常现象。
- 主线没有塌。
- anchor 参数没有明显漂移。
- 说明收紧执行口径后，这套系统依旧成立。

这个阶段非常重要，因为它证明了：

- 系统不是靠理想化成交价硬撑出来的。

### 3.6 阶段 F：加入本金与杠杆的金额化表达

用户明确提出，希望回测与纸面组合不要只看收益率，而要用更直观的本金口径表达。

最终确定的金额化方案是：

- 每个交易对 `1000U`
- 实际观察组合按 `BTC + ETH` 先看，总本金 `2000U`
- 历史表达中可扩展到更多币种
- 使用 `2x leverage`

为此，修改了：

- `analyze_breakout_trade_outcomes.py`

新增能力：

- `--initial-capital-usd`
- `--leverage`

输出中增加：

- `entry_equity_usd`
- `exit_equity_usd`
- `gross_pnl_usd`
- `net_pnl_usd`

这个阶段的意义是：

- 把系统从“抽象百分比模型”推进成“更像真实账户”的表达方式。

### 3.7 阶段 G：从静态结果页到动态中文看板

系统进入工程化后，用户明确提出：

- 需要一个动态 Web 页面
- 能实时展示买入、持仓、平仓等综合信息
- 不希望手动刷新
- 页面要改成简体中文，方便观察

于是新建了：

- `mnts_paper_dashboard.py`

第一版实现内容包括：

- Flask Web 服务
- 自动刷新
- BTC / ETH 卡片
- 历史回测基线
- 实时状态展示

后续逐步改良：

- 页面全部中文化
- `long/short` 改成 `多头/空头`
- 指标名、表头、提示文案全部改成更直观的中文

这一步里有一个很典型的交互问题：

- 页面里一开始显示 `0 / 39`
- 用户无法直接理解

后来经过两轮优化，最终改成：

- `当前状态：断裂，已持续 39 根`
- `当前状态：中性漂移`

这个优化非常小，但意义很大：

- 它把系统从“开发者能看懂”推进到了“观察者一眼能读懂”。

### 3.8 阶段 H：从看板展示升级为连续 paper ledger

用户随后追问：

- 页面到底展示的是历史结果，还是已经开始观察未来？

这推动系统从“展示页”升级成“连续纸上账户”。

新增了本地持久化账本目录：

- `runtime/paper_trading/btcusdt/`
- `runtime/paper_trading/ethusdt/`

包含文件：

- `paper_account_state.json`
- `signal_log.csv`
- `closed_trades.csv`
- `equity_curve.csv`
- 后来加入 `model_cycle_state.json`

页面新增了：

- 实时账户初始本金
- 实时账户已实现权益
- 实时账户已实现净盈亏
- 实时账户当前浮盈亏
- 实时信号流水
- 实时纸面已平仓交易

同时明确修正了首次启动口径：

- `第一次启动只从现在开始记账`
- `不回放旧历史`

这样避免了一个很常见的工程误区：

- 纸上账户一启动就把最近窗口的历史信号倒灌到账本里

修正后，这套观察系统才真正符合“从现在开始观察未来”的意图。

### 3.9 阶段 I：从轻量实时重训切换到严格 walk-forward 观察版

这是整个对话里最关键、也最容易走偏的一步。

最初的实时链路是：

- 拉最近几百根 `15m` K 线
- 现场做 tokenizer
- 现场拟合一个 V2 模型
- 直接算最新状态和持仓

这条链路能跑通，但用户很快指出一个本质问题：

- 这和之前的历史回测口径并不一致

因为之前的历史回测不是“每 15m 实时重训”，而是：

1. 拿 `180天` 做训练
2. 跑后面 `45天` 测试
3. `45天` 跑完以后，窗口整体前移
4. 再重新训练一次

于是后续正式把看板底层改成：

- `180天训练`
- `45天冻结运行`
- `到期再滚动重训`

这一步把实时观察系统从“实验型在线拟合”切成了“回测同思路的冻结模型周期版”。

### 4.4 当前模型更新口径

- `180天训练`

### 4.5 当前特征统计口径

最初切成冻结模型周期后，仍然留了一个细微偏差：

- `prepare_features()` 中 `embedding_anomaly` 的 `center / scale`
- 会混入整段周期切片中的运行窗数据

这意味着：

- 外层模型虽然冻结了
- 但特征参考系没有完全冻结

后来进一步修正为：

- 训练窗拟合 `center / scale`
- 运行窗只应用训练窗拟合出来的统计基准

并在模型状态文件中明确标记：

- `feature_fit_scope = train_window_only`

这一步把实时观察系统从“方向一致”进一步推进到“口径也尽量干净一致”。

## 5. 本轮开发中遇到的核心问题与解决方案

下面按问题类型归纳。

### 5.1 问题一：实时轻量版与原回测口径不一致

#### 表现

- 页面能动态展示
- 实时链路能跑通
- 但底层是“最近若干根 K 线现场拟合 V2”

#### 风险

- 这会让观察版和原 `180/45 walk-forward` 回测产生方法论偏移
- 看起来很像实盘，但其实不是我们验证过的那种系统

#### 解决

把 `mnts_paper_dashboard.py` 底层改为：

- `180天训练`
- `45天冻结运行`
- `到期再重训`

并新增模型周期元数据展示：

- 当前训练区间
- 当前运行区间
- 下次重训时间
- 运行进度

#### 结果

- 动态看板的底层逻辑与历史主回测思路重新对齐

### 5.2 问题二：特征统计基准仍带轻微前视偏差

#### 表现

虽然已经切成冻结模型周期，但 `prepare_features()` 中：

- `embedding_anomaly`

默认还是用整段输入一起算 `center / scale`

#### 风险

- 训练窗之外的运行窗分布被提前混进特征参考系
- 形成轻微前视偏差和轻微乐观化风险

#### 解决

把特征准备拆成两步：

1. 训练窗拟合统计参数
2. 运行窗只应用统计参数

#### 结果

- 特征参考系也被一起冻结
- 更严格地符合 walk-forward 精神

### 5.3 问题三：旧模型周期文件会沿用旧口径

#### 表现

即使代码改了，如果：

- `runtime/paper_trading/*/model_cycle_state.json`

已经存在，那么系统会继续复用旧模型文件。

#### 风险

- 代码逻辑和磁盘模型状态不一致
- 页面看起来正常，实际却还在使用上一版口径

#### 解决

在模型缓存判定里增加：

- `feature_fit_scope == train_window_only`

如果不满足，自动失效并重训。

#### 结果

- 新旧口径不会再混用

### 5.4 问题四：首次启动 paper ledger 时会补记旧历史

#### 表现

初版连续账本会把最近窗口里的历史信号直接补进账本。

#### 风险

- 这不是“从现在开始观察”
- 而是“带历史回放的伪观察”

#### 解决

加入首次启动修正：

- 如果账本 state 不存在
- 则把 `last_processed_signal_timestamp` 直接设为当前最后一条 ledger row
- 不补回放过去历史

#### 结果

- 纸上账户真正从当前时刻开始累计

### 5.5 问题五：页面状态文案对用户不友好

#### 表现

- `cohesion_streak / fracture_streak`
- 以 `0 / 39` 的形式显示

#### 风险

- 观察者看不出含义
- 看板虽然有数据，但不具备可读性

#### 解决

逐步把文案改成：

- `当前状态：断裂，已持续 39 根`
- `当前状态：中性漂移`

#### 结果

- 页面从“开发视角可读”变成“观察视角可读”

### 5.6 问题六：旧服务占口，导致页面看起来没更新

#### 表现

- 代码里已经新增新字段
- 但 `/api/state` 里还是旧结构
- 或者重启时出现 `Address already in use`

#### 风险

- 容易误判为代码没生效
- 实际上是旧进程还占着端口

#### 解决

反复通过：

- `ps -ef | grep mnts_paper_dashboard.py`
- `ss -ltnp '( sport = :8787 )'`

定位旧 PID 并清理后再启动。

#### 结果

- 代码、服务、API 输出重新对齐

### 5.7 问题七：预热前接口首屏过慢

#### 表现

切到严格周期后，首轮实时快照计算非常重。

#### 风险

- `/api/state` 首次请求等待极长
- 页面首屏卡死

#### 解决

做了两层收紧：

1. 不再对整段 4 年历史全量 tokenizer，而只对“当前活跃周期切片”做 tokenizer。
2. 服务启动时先预热 live snapshot，再开始监听。

同时调整缓存失效机制：

- 不是固定每分钟过期
- 而是尽量延到下一根可能确认的 `15m` bar 再更新

#### 结果

- 预热后 `/api/state` 响应恢复到亚秒级

### 5.8 问题八：实时信号开仓价不能从最近 live bars 反推

#### 表现

如果持仓时间较长，只从最近 live K 线窗口里反推：

- 当前仓位的入场价可能会丢失或取错

#### 解决

直接从当前冻结运行窗的信号行里读取：

- `next_open_fill_price`
- `next_fill_timestamp`

#### 结果

- 当前持仓的浮盈亏、参考入场价计算更稳定，更符合账本口径

## 6. 本轮做过的重要验证

### 6.1 多品种泛化验证

验证对象：

- BTC
- ETH
- SOL
- XRP
- DASH

结论：

- 主线具有跨品种迁移能力
- 但不同品种的流动性与回撤特征差异明显

### 6.2 参数稳定性验证

验证目标：

- 确认不是只踩中某一个历史最优点

结论：

- 存在稳定参数区
- 适合选 anchor 参数，而不是追峰值

### 6.3 执行口径收紧验证

验证目标：

- `close_to_next_open`
- `fee4 + slippage1`

结论：

- 强度下降但主线不塌
- anchor 参数没有显著漂移

### 6.4 当前版与严格特征冻结版偏差验证

在修正特征统计基准之前，专门比较了：

- 当前版
- 严格训练窗统计参数版

在当时活跃运行窗中的差异。

实测结果显示：

- `BTC` 当前运行窗内 `state_diff_bars = 0`
- `BTC` 当前运行窗内 `position_diff_bars = 0`
- `ETH` 当前运行窗内 `state_diff_bars = 0`
- `ETH` 当前运行窗内 `position_diff_bars = 0`

结论：

- 那个偏差在当前窗口里影响极小
- 但出于系统洁净度要求，仍然应该修掉

### 6.5 Web/API 运行态验证

验证方式包括：

- `build_api_state()` 本地执行
- `py_compile`
- 真正启动 Flask 服务
- 轮询 `/api/state`

最终确认：

- 服务能正常启动
- 模型周期字段能正常返回
- 预热完成后接口响应速度恢复到可用水平

## 9. 当前系统已经完成到什么程度

如果用“研究 -> 收敛 -> 观察 -> 真仓”四阶段来分，目前处于：

- `观察阶段`

更准确地说，是：

- `第一版严格口径纸上观察阶段`

当前已经具备：

- 固定主线
- 固定主样本
- 固定 anchor
- 严格周期更新逻辑
- 严格特征统计基准冻结
- 金额化表达
- 持续账本
- 动态 Web 观察界面

当前还没有做的，是：

- 真实交易所下单执行
- 更复杂的盘口冲击和资金费率建模
- 正式实盘风险控制制度

所以它现在的准确定位是：

- 不是研究实验
- 不是历史展示
- 也不是实盘
- 而是“未来真实时间里的严格纸面观察系统”

## 12. 下一阶段建议

当前最合理的下一阶段，不是继续大规模改代码，而是：

- 正式进入 `4-8 周` 纸上观察期

观察期重点关注：

- 当前持仓状态是否稳定

- 是否把 `SOL / XRP` 接入观察
